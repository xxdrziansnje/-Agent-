================================================================================
  智扫通机器人智能客服 — 项目架构分析
================================================================================

【项目概述】
基于 LangGraph ReAct Agent 的智能客服系统，使用 Streamlit 做前端、
通义千问(qwen3-max)做推理、Chroma 做向量检索，模拟扫地机器人品牌的
客服问答和使用报告生成场景。

================================================================================
  二、核心交互流程 — 时序图（Mermaid）
================================================================================

sequenceDiagram
    actor User as 用户
    participant UI as Streamlit app.py
    participant Agent as ReactAgent
    participant MW as Middleware
    participant LLM as qwen3-max
    participant Tool as Tools
    participant RAG as RAG服务
    participant DB as Chroma

    Note over User,DB: === 流程A: 普通问答 ===

    User->>UI: 输入问题
    UI->>Agent: execute_stream(query)
    Agent->>Agent: 构建输入 {messages:[HumanMessage]}
    Agent->>LLM: stream(context={report:false})
    activate LLM
    MW-->>LLM: log_before_model(记录调用)
    LLM->>LLM: ReAct推理，决定调用工具
    LLM->>Tool: 执行工具 (如 rag_summarize)
    MW-->>Tool: monitor_tool(记录前后)
    Tool->>RAG: 检索相关文档
    RAG->>DB: similarity_search(k=3)
    DB-->>RAG: Top-3 文档片段
    RAG->>LLM: LCEL链: 格式化->LLM->摘要
    LLM-->>Tool: 返回结果
    Tool-->>LLM: 工具结果
    deactivate LLM
    LLM->>LLM: 综合结果生成最终回答
    LLM-->>Agent: 流式输出字符块
    Agent-->>UI: yield 字符块(+10ms延迟)
    UI-->>User: 打字机效果渲染

    Note over User,DB: === 流程B: 报告生成 ===

    User->>UI: "帮我生成使用报告"
    UI->>Agent: execute_stream(...)
    Agent->>LLM: stream(context={report:false})
    LLM->>Tool: get_user_id() -> get_current_month()
    LLM->>Tool: fill_context_for_report()
    MW-->>MW: monitor_tool 捕获, 设置 context.report=true
    MW-->>LLM: report_prompt_switch 替换为 report_prompt.txt
    LLM->>Tool: fetch_external_data(uid, month)
    LLM->>Tool: rag_summarize(保养建议)
    LLM->>LLM: 生成 Markdown 报告
    LLM-->>UI: 流式输出完整报告

  三、提示切换机制 — 状态图（Mermaid）
================================================================================

stateDiagram-v2
    [*] --> NormalMode: 初始 context.report=false
    NormalMode --> NormalMode: 加载 main_prompt.txt
    NormalMode --> TriggerCheck: Agent决定调用 fill_context_for_report()
    TriggerCheck --> MiddlewareCapture: monitor_tool 中间件捕获
    MiddlewareCapture --> ContextSwitch: 设置 context["report"] = True
    ContextSwitch --> ReportMode: 下一次 LLM 调用前
    ReportMode --> ReportMode: report_prompt_switch 切换加载 report_prompt.txt

    NOTE: fill_context_for_report 工具本身无实际业务逻辑，仅作为"开关"触发器

================================================================================
  七、一句话总结
================================================================================

LangGraph ReAct Agent 封装了 7 个工具和 3 个中间件，通过 fill_context_for_report
工具 + monitor_tool 中间件 + report_prompt_switch 中间件实现了运行时动态提示切换，
从而在"普通客服问答"和"使用报告生成"两种模式间无缝切换。RAG 层用 Chroma 做向量检索，
通义千问做摘要生成，全链路支持流式输出。

================================================================================
  八、关键设计要点
================================================================================

1. 提示切换 (Middleware-Driven)
   fill_context_for_report 工具存在的唯一目的是在运行时上下文中设置布尔标志，
   然后由 report_prompt_switch 中间件读取以交换全部系统提示。
   这是一种基于上下文的提示路由模式。

2. 模拟数据层
   所有数据存储端点（天气、位置、用户ID、月份、使用记录）当前返回硬编码/随机值。
   records.csv 是唯一包含真实模拟数据的文件，涵盖10用户×12月数据。

3. 双重 Chroma 数据库
   agent/chroma_db/ — Agent层数据库
   rag/chroma_db/    — RAG层数据库，动态填充自 data/ 目录

4. MD5 去重
   md5.text 作为简单注册表，防止文档被重复加载到向量存储中。

5. 导入时单例
   所有核心单例（chat_model, embed_model, rag_conf, chroma_conf,
   prompts_conf, agent_conf, logger）均在导入时创建，无需显式初始化。

6. 流式输出
   Streamlit 端通过 capture() 内部函数缓冲字符块，增加10ms字符间延迟
   实现打字机效果渲染。
